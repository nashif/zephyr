.. _process_model_build:

Building and Running Processes
##############################

This page describes how to set up a Zephyr project that uses the Process Model,
how to build extension ELF binaries, and how to embed and load them at runtime.

Project Setup
*************

Kconfig
=======

Enable the Process Model by adding the following to your ``prj.conf``:

.. code-block:: kconfig

   CONFIG_LLEXT=y
   # Set the LLEXT heap large enough to hold all extensions loaded at once
   CONFIG_LLEXT_HEAP_SIZE=16   # kilobytes

   CONFIG_PROCESS=y

   # Recommended: enable full memory isolation
   CONFIG_USERSPACE=y

   # Increase if you need more than the default 8 domain partitions
   # (LLEXT uses up to 4 per process: TEXT, DATA, RODATA, BSS)
   CONFIG_MAX_DOMAIN_PARTITIONS=8

Key Kconfig symbols introduced by the Process Model subsystem:

:kconfig:option:`CONFIG_PROCESS`

   Master switch.  Depends on :kconfig:option:`CONFIG_LLEXT`.

:kconfig:option:`CONFIG_PROCESS_NAME_MAX_LEN`

   Maximum length for a process name, not counting the null terminator.
   Default: ``15``.

:kconfig:option:`CONFIG_PROCESS_STACK_SIZE_DEFAULT`

   Stack size (bytes) used when :c:member:`z_process_opts.stack_size` is ``0``.
   Default: ``2048``.

:kconfig:option:`CONFIG_PROCESS_PRIORITY_DEFAULT`

   Scheduling priority for newly created process threads.  Default: ``5``.

:kconfig:option:`CONFIG_PROCESS_LOG_LEVEL`

   Log level for the ``process`` module.  Set to ``4`` (``DBG``) to trace
   every state transition.

CMakeLists.txt
==============

Building an extension ELF binary from source is done with the
``add_llext_target()`` CMake helper provided by the Zephyr build system, and
embedding it as a C array is done with ``generate_inc_file_for_target()``:

.. code-block:: cmake

   cmake_minimum_required(VERSION 3.20.0)
   find_package(Zephyr REQUIRED HINTS $ENV{ZEPHYR_BASE})
   project(my_app LANGUAGES C)

   # ── Build the extension ELF ─────────────────────────
   add_llext_target(my_ext
     OUTPUT  ${PROJECT_BINARY_DIR}/llext/my_ext.llext
     SOURCES ext/my_ext.c
   )

   # ── Embed the ELF as a C array ──────────────────────
   generate_inc_file_for_target(
     app
     ${PROJECT_BINARY_DIR}/llext/my_ext.llext
     ${ZEPHYR_BINARY_DIR}/include/generated/my_ext.inc
   )

   # ── Host application ────────────────────────────────
   target_sources(app PRIVATE src/main.c)

Writing the Extension
*********************

An extension is a normal C file compiled with the LLEXT toolchain.  It must
export its entry point using :c:macro:`LL_EXTENSION_SYMBOL`:

.. code-block:: c

   /* ext/my_ext.c */
   #include <zephyr/kernel.h>
   #include <zephyr/llext/symbol.h>

   void process_main(void *arg)
   {
       /* Extension code runs here.
        * When CONFIG_USERSPACE is enabled this executes in user mode:
        * only the stack and the extension's own TEXT/DATA/RODATA/BSS
        * are accessible.
        */
       printk("Hello from the extension! arg=%p\n", arg);
   }

   LL_EXTENSION_SYMBOL(process_main);

The entry symbol name can be anything; set
:c:member:`z_process_opts.entry_sym` to override the default
(``"process_main"``).

Writing the Host Application
*****************************

.. code-block:: c

   /* src/main.c */
   #include <zephyr/kernel.h>
   #include <zephyr/process/process.h>

   /* Embedded ELF binary (generated by the build system) */
   static const uint8_t my_ext_elf[] = {
   #include <my_ext.inc>
   };

   /* Stack for the process thread */
   Z_PROCESS_STACK_DEFINE(my_stack, CONFIG_PROCESS_STACK_SIZE_DEFAULT);

   /* Process descriptor (global or static) */
   static struct z_process my_proc;

   int main(void)
   {
       /* Optional: customise options */
       struct z_process_opts opts = Z_PROCESS_OPTS_DEFAULT;
       opts.arg = (void *)(uintptr_t)42;   /* forwarded to process_main */

       /* Load the ELF, create the domain, resolve the entry symbol, and
        * start the thread — all in one call. */
       int ret = z_process_spawn(&my_proc, "my_ext",
                                 my_ext_elf, sizeof(my_ext_elf),
                                 my_stack, sizeof(my_stack),
                                 &opts);
       if (ret != 0) {
           printk("spawn failed: %d\n", ret);
           return ret;
       }

       /* Wait for the process thread to exit */
       z_process_join(&my_proc, K_FOREVER);

       /* Free LLEXT memory and reset the descriptor */
       z_process_unload(&my_proc);
       return 0;
   }

Building and Running the Samples
*********************************

Hello World Sample
==================

The :zephyr_file:`samples/subsys/process/hello` sample demonstrates the
minimal load → spawn → join → unload cycle.

.. zephyr-app-commands::
   :zephyr-app: samples/subsys/process/hello
   :board: qemu_cortex_r5
   :goals: build run
   :compact:

Expected output:

.. code-block:: console

   [00:00:00.000,000] <inf> main: Process Model sample: hello world
   Hello from process 'hello' (arg=42)
   [00:00:00.000,000] <inf> main: Process exited with code 0
   [00:00:00.000,000] <inf> main: Done

To run without memory protection (supervisor mode only):

.. zephyr-app-commands::
   :zephyr-app: samples/subsys/process/hello
   :board: qemu_cortex_r5
   :goals: build run
   :gen-args: -DCONFIG_USERSPACE=n
   :compact:

Isolation Sample
================

The :zephyr_file:`samples/subsys/process/isolation` sample runs two
independent processes concurrently in separate memory domains.

.. zephyr-app-commands::
   :zephyr-app: samples/subsys/process/isolation
   :board: qemu_cortex_r5
   :goals: build run
   :compact:

Expected output (order of A/B lines may vary):

.. code-block:: console

   [00:00:00.000,000] <inf> main: Process Model sample: isolation
   Process A: counter=1
   Process B: counter=5
   Process A: counter=2
   Process B: counter=4
   Process A: counter=3
   Process B: counter=3
   Process A: counter=4
   Process B: counter=2
   Process A: counter=5
   Process B: counter=1
   [00:00:00.000,000] <inf> main: All processes finished
   [00:00:00.000,000] <inf> main:   proc_a exit code: 0
   [00:00:00.000,000] <inf> main:   proc_b exit code: 0

Running on Other Boards
========================

Any board with an architecture supported by LLEXT and sufficient RAM can be
used.  To run on ``mps2/an385`` (Cortex-M3/ARMv7-M):

.. zephyr-app-commands::
   :zephyr-app: samples/subsys/process/hello
   :board: mps2/an385
   :goals: build run
   :compact:

.. note::

   On ARMv7-M, the MPU requires regions to be power-of-two aligned and sized.
   Increase :kconfig:option:`CONFIG_LLEXT_HEAP_SIZE` to at least ``16`` KB if
   loading fails with ``-ENOMEM`` to provide the allocator with enough room to
   satisfy the stricter alignment requirements.

Running the Tests
*****************

The test suite lives in :zephyr_file:`tests/subsys/process` and covers the
full API surface including lifecycle, argument forwarding, concurrent
processes, kill, and error paths.

To run the tests with Twister:

.. code-block:: bash

   ./scripts/twister -T tests/subsys/process \
       -p qemu_cortex_r5 \
       -p mps2/an385 \
       -p mps2/an521/cpu0

Two variants are defined in
:zephyr_file:`tests/subsys/process/testcase.yaml`:

``subsys.process.lifecycle``
   Tests run with :kconfig:option:`CONFIG_USERSPACE` enabled.  Process threads
   execute in unprivileged mode and the MPU/MMU enforces isolation.

``subsys.process.no_userspace``
   Tests run with :kconfig:option:`CONFIG_USERSPACE` disabled.  All threads
   run in supervisor mode.  The ``test_arg_forwarding`` case is fully
   asserted since the extension can write to kernel memory.

Common Configuration Pitfalls
*******************************

.. list-table::
   :widths: 40 60
   :header-rows: 1

   * - Symptom
     - Solution
   * - ``z_process_load`` returns ``-ENOMEM``
     - Increase :kconfig:option:`CONFIG_LLEXT_HEAP_SIZE`.  On MPU platforms
       with power-of-two alignment requirements, more space is needed than the
       raw extension size suggests.
   * - ``z_process_load`` returns ``-ENOENT``
     - The entry symbol was not found in the extension.  Check that
       ``LL_EXTENSION_SYMBOL(process_main)`` is present in the extension source
       and that ``opts.entry_sym`` matches the exported name.
   * - User-mode thread receives a memory fault immediately on entry
     - A kernel object (semaphore, message queue, etc.) passed via
       ``opts.arg`` was not granted to the thread.  Call
       :c:func:`k_object_access_grant` before :c:func:`z_process_start`.
   * - ``CONFIG_MAX_DOMAIN_PARTITIONS`` exceeded
     - Each LLEXT process uses 4 partitions (TEXT, DATA, RODATA, BSS).
       Increase ``CONFIG_MAX_DOMAIN_PARTITIONS`` accordingly
       (e.g., ``8`` for two concurrent processes).
   * - Build fails: ``add_llext_target not defined``
     - Ensure :kconfig:option:`CONFIG_LLEXT` is enabled in ``prj.conf``.
